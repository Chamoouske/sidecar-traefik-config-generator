# Context — Traefik Sidecar

## Glossary

### Cross-node routing
Mechanism that allows a request arriving at Traefik on Node A to be forwarded to a service running on Node B, by routing through Node B's Traefik instance via the host IP address.

### Agent
Per-node sidecar component. Mounts the Docker socket to discover local containers and their `traefik.sidecar.*` labels. Connects to all other Agents in a full mesh via gRPC bidirectional streams, exchanging container reports. Each Agent independently computes and writes its own Traefik configuration files.

### Service
A Docker container with `traefik.sidecar.*` labels. The sidecar is the sole source of Traefik routing configuration — services never carry standard `traefik.http.*` labels.

### Node
A machine running Docker Engine (standard mode, not Swarm).

### Host IP
The routable IP address of a node's network interface, used by cross-node routes to forward traffic between nodes via their host network.

### Local route
A Traefik route for a container running on the same node. Generated by the sidecar with weight 9.

### Cross-node route
A Traefik configuration that forwards traffic from one node's Traefik to another node's Traefik (via the host IP), for services that need to be reachable across all nodes. Generated with weight 1.

### Weighted routing
Mechanism where local routes receive higher weight (9) than cross-node routes (1), proportionally distributing traffic toward local containers and exponentially reducing the probability of forwarding loops.

### Safety net polling
Periodic full-state exchange between Agents (default: 60s) that verifies all peer container info is accurate. Acts as a recovery mechanism in case a gRPC notification was missed.

### Full mesh
Topology where every Agent maintains a direct gRPC connection to every other Agent in the cluster. A new Agent discovers existing peers via mDNS and establishes connections immediately.

### mDNS (multicast DNS)
Zero-configuration discovery protocol used by Agents to find each other on the local network. Each Agent announces its service type (`_traefik-sidecar._tcp`) and listens for peer announcements. Limited to a single LAN segment; does not work across subnets, WAN, or out of Docker Desktop's VM.

### Static peer
A peer configured explicitly via `TRAEFIK_SIDECAR_PEERS` environment variable. Used when mDNS is unavailable (Docker Desktop, cross-subnet). Bypasses the hostname comparison guard — both sides always dial.

### Peer
Another Agent instance in the mesh, discovered via mDNS or configured statically.

### gRPC bidirectional stream
Persistent connection between two Agents, used for real-time exchange of `ContainerReport` messages. Each Agent sends its local container list and receives peer container lists on the same stream.

### ContainerReport
A protobuf message containing all containers running on a node, including their `traefik.sidecar.*` labels, exposed ports, and network settings. Sent by an Agent to all its peers whenever a local container starts, stops, or changes.

### Cross-node transport
The TLS connection between two Traefik instances for cross-node routing. By default it uses HTTP/1.1 (`forwardHTTPVersion: false`), which preserves WebSocket upgrades. Services that need HTTP/2 (e.g. gRPC) must opt in via `traefik.sidecar.service.http2=true`, which selects the `sidecar-internal-h2` transport with `forwardHTTPVersion: true` so the client's HTTP/2 connection is preserved end-to-end.

### Middleware
A Traefik middleware definition, configured via `traefik.sidecar.middleware.<name>.<type>.<option>` labels on a container. Middlewares are collected across all containers and written to a global `_middlewares.yml` file. Routers reference them via `traefik.sidecar.router.middlewares`.

### NodeConfig
The complete set of Traefik routers, services, and middlewares for a single node, computed by each Agent from its combined local + peer container information.
