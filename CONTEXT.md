# Context — Traefik Sidecar

## Glossary

### Cross-node routing
Mechanism that allows a request arriving at Traefik on Node A to be forwarded to a service running on Node B, by routing through Node B's Traefik instance via the host IP address. Previously called "federation".

### Hub
Central coordinator component running on the Swarm manager node. Reads the Docker/Swarm API to maintain global state of all services and tasks, and instructs agents on which cross-node routes to create.

### Agent
Per-node sidecar component. Initiates a gRPC bidirectional stream connection to the Hub, receives configuration commands, and writes local Traefik configuration files. Does not mount the Docker socket.

### Service
A Docker Swarm service with `traefik.sidecar.*` labels. The sidecar system is the sole source of Traefik routing configuration — services never carry standard `traefik.http.*` labels.

### Task
A running instance (replica) of a Service on a specific Swarm node.

### Node
A machine (Linux or Windows) participating in the Docker Swarm cluster.

### Local route
A Traefik route served on the same node where the task is running. Generated by the sidecar, not by the Swarm provider.

### Cross-node route
A Traefik configuration that forwards traffic from one node's Traefik to another node's Traefik (via the host IP), for services that need to be reachable across all nodes.

### Weighted routing
Mechanism where local routes receive higher weight than cross-node routes, proportionally distributing traffic toward local tasks and preventing routing loops.

### Safety net polling
Periodic check performed by the Agent to the Hub, verifying that existing cross-node routes are still valid. Acts as a recovery mechanism in case the Hub failed to notify a route removal.

### gRPC bidirectional stream
Persistent connection initiated by the Agent to the Hub, used for real-time command delivery (Hub → Agent) and status reporting (Agent → Hub).
